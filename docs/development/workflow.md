# 開発運用ルール

この文書は、このリポジトリで開発作業を進めるときの標準手順をまとめる。Codex や別の生成 AI が新しいコンテキストで作業する場合も、まずこの文書と [project_context.md](project_context.md) を読む。

## 基本方針

- ユーザーへの応答、GitHub Issue、Pull Request は日本語で書く。
- GitHub で管理する作業は必ず Issue に紐づける。
- 原則として 1 Issue = 1 PR とし、複数 Issue をまとめて閉じる大きな PR は避ける。
- Issue は実装者が判断に迷わない粒度まで分割し、背景、対応内容、受け入れ条件、テスト方針、対象外を明記する。
- 作業ブランチは `codex/` prefix を基本にし、ブランチ名に Issue 番号と要約を入れる。
- Codex は PR を作成してよいが、マージは人間が行う。Codex は絶対にマージしない。

## Issue の切り方

Issue は、実装・レビュー・テストが独立して判断できる単位にする。

- 低層 I/O、ファイル形式、コマンド統合、実機確認は分ける。
- バグ修正は機能追加から分ける。たとえば `CMD_RESUME` 修正は SD/FAT 実装とは別 Issue にする。
- 受け入れ条件は「動くこと」ではなく、入力、期待出力、確認方法が分かる形にする。
- 親 Issue を使う場合でも、サブ Issue の PR から親 Issue を自動 close しない。本文では `Refs #番号` を使う。
- PoC や調査ブランチを main に直接統合しない。取り込む場合は、必要な変更を小さい Issue/PR に分け直す。

## PR の作り方

PR は対応 Issue 単位で作る。PR 本文には次を必ず書く。

- 対応 Issue。
- 変更内容。
- テスト結果。
- 追加または更新したテスト。
- 影響範囲。
- 未対応事項。
- 関連リンク。

PR 作成前に、差分へ不要ファイルが入っていないことを確認する。

- `.DS_Store` を含めない。
- 実験用の巨大イメージや一時生成物を含めない。
- 個人メモや `wk_問答.md` を混ぜない。
- Git 管理対象外の `blog_work/` を含めない。
- PoC ブランチ由来の不要ファイルをそのまま持ち込まない。

## テスト方針

PR 作成前に、原則として次を実行する。

```powershell
make bin
$env:REQUIRE_BUILD_ROM='1'
python tests/test_smoke.py
```

実装で既存 smoke test にない振る舞いを追加・変更した場合は、対応するテストを追加する。

- 既存コマンドを変更したら、既存コマンドの回帰テストを追加または更新する。
- 新規コマンドを追加したら、正常系と最低限の異常系を smoke test に追加する。
- エミュレータや fixture を使う場合は、固定入力と期待結果を明示する。
- 実機でしか確認できない内容は、手順と確認結果を `docs/testing/` または `docs/progress/` に残す。

CI では `REQUIRE_BUILD_ROM=1` を使い、最新ソースから生成した `build/mc6800-monitor.bin` を検証する。fixture はローカル補助用であり、最新ソースの確認には使わない。

## レビュー運用

- 作業は基本的に `実装計画`、`実装者`、`批判的レビュー者` の役割を分ける。
- 批判的レビュー者は、バグ、回帰、未テスト、要件ズレ、不要ファイル混入を優先して確認する。
- 批判的レビュー者から指摘が出た場合は、まずユーザーへ共有して対応判断を仰ぐ。
- レビュー指摘をユーザー確認なしで握りつぶさない。
- 指摘対応後は、どの指摘をどう反映したかをPR本文またはコメントに残す。

## ドキュメント運用

- 要件変更は `docs/requirements/` に残す。
- 設計判断やメモリマップは `docs/design/` に残す。
- 実装順序やフェーズ分割は `docs/plans/` に残す。
- 実機確認手順や検証観点は `docs/testing/` に残す。
- 作業の区切りや実機確認結果は `docs/progress/YYYY-MM-DD.md` に残す。
- 開発手順や引き継ぎコンテキストは `docs/development/` に残す。

Blog は補助資料として扱う。開発に必要な判断、制約、Issue/PR へのリンクは Git 管理下の docs にも残す。
