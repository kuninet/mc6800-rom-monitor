# 進捗ログ 2026-04-14

## 実施内容

- `L` コマンドから入るローダモードを実装した
- ローダ中は通常 CLI の `READ_LINE` を使わず、無エコーの `READ_RECORD` で 1 レコードずつ受信する形にした
- 受信した先頭レコードから S-Record と Intel HEX を自動判別するようにした
- S-Record と Intel HEX のチェックサム検証を実装した
- 正常終了時に `OK`、異常時に `?S1` 〜 `?S5` / `?I1` 〜 `?I5` を返す保守用表示を追加した
- `M` コマンドの値入力を 1〜2 桁だけ許可し、3 桁以上を `?` にするよう修正した

## 実機確認

- SBC6800 実機で S-Record の 3 行サンプルをロードし、`OK` を確認した
- SBC6800 実機で Intel HEX の 3 行サンプルをロードし、`OK` を確認した
- `D0100` でロード内容をダンプし、期待したデータが RAM に配置されていることを確認した
- Tera Term からの貼り付けでは、少なくとも現状の 9600bps / フロー制御なし構成では行間待ちを入れた方が安定することを確認した

## 判明したこと

- S-Record ではデータループ中に解析用 `X` を書き込み先アドレスで上書きしており、2 バイト目以降で失敗する不具合があった
- Intel HEX ではヘッダ長チェック後に失敗ラベルへ落ちる分岐ミスがあり、正常レコードでも `?I2` になる不具合があった
- デバッグ表示 `?S4` / `?I2` により、実機上でも失敗箇所の切り分けが大幅にやりやすくなった

## 既知の制限

- 初版ローダは 16bit アドレス空間前提で、Intel HEX 拡張アドレスや S-Record `S3/S7` は未対応
- 終端レコード検証は最小実装であり、Intel HEX EOF や S-Record EOF の厳密条件までは検証していない
- チェックサム不正時は、当該レコードの一部データが先に RAM へ書き込まれている可能性がある
- ローダはフロー制御を持たないため、高速貼り付けでは行間待ちが必要になる場合がある
